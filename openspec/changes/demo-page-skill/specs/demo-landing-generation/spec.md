# Delta for demo-landing-generation

> Deliverable note: the future `skills/demo-landing/SKILL.md` and all demo UI copy
> MUST be Mexican Spanish, formal "usted" (project convention). This spec, an SDD
> artifact, stays English.

## ADDED Requirements

### Requirement: Real-Data-Only Content

The generator MUST render only content traceable to a field in the prospect's real
JSON record. It MUST NOT invent sector claims, statistics, services, or contact data,
and no rendered stat may contradict the record.

#### Scenario: Every claim maps to a field

- GIVEN a prospect record with `empresa`, `problemas_detectados`, `oportunidades`
- WHEN the demo is generated
- THEN every factual claim about the prospect maps to a value in that record, and
  sector-typical Tier B copy carries no fabricated specifics (scopes, prices,
  credentials, or capabilities the record does not back)
- AND no stat asserts "HTTPS" when `https:false`

#### Scenario: Missing optional field

- GIVEN a prospect whose `email` and `whatsapp` are null (e.g. `id:2`)
- WHEN the demo is generated
- THEN blocks depending on those fields are omitted, not filled with placeholders

### Requirement: Required Sections

The landing MUST contain seven sections — navbar, hero, about, why-us, services, FAQ,
footer — built with vanilla HTML/CSS/JS plus animations and gradients.

#### Scenario: All seven sections present

- GIVEN a top-10 prospect
- WHEN the demo HTML is produced
- THEN navbar, hero, about, why-us, services, FAQ, and footer are all present
- AND why-us derives from `problemas_detectados`/`oportunidades` framed as improvements
  the mockup demonstrates, while services use sector-typical Tier B copy plus real
  Tier A-backed capabilities — never the seller's `oportunidades` upsells as if they
  were the prospect's own services

### Requirement: Brand-First Palette with Noted Fallback

The generator MUST derive the palette from the prospect's existing site when it yields
≥2 usable colors at ≥4.5:1 contrast; otherwise it MUST fall back to a sector palette
(ui-ux-pro-max) and MUST record that fallback in the campaign report.

#### Scenario: Brand extraction succeeds

- GIVEN a prospect with a live site (`tiene_web:true`)
- WHEN colors are extracted and ≥2 pass the contrast gate
- THEN the demo uses the brand palette

#### Scenario: Fallback to sector palette

- GIVEN extraction yields <2 usable colors, or the prospect has no site
- WHEN the demo is generated
- THEN a sector-appropriate palette is used
- AND the campaign report notes the palette fallback for that prospect

### Requirement: Self-Contained Local Assets

The generator MUST download CC0 images into
`tracking/demos-publicar/assets/[tracking_id]/` and reference them by relative path, so
a deployed demo issues no external request except the tracking worker.

#### Scenario: Images stored per demo

- GIVEN a demo for `tracking_id` `contadores-cdmx-01`
- WHEN generation completes
- THEN its images live under `assets/contadores-cdmx-01/` referenced by relative path

#### Scenario: Image download fails verification

- GIVEN an image fails MIME/size verification
- WHEN the demo is generated
- THEN a gradient/SVG fallback is used and no broken image ships

### Requirement: Tracking Pixel and Real-Data CTAs

The demo MUST embed the tracking pixel and CTAs (WhatsApp, email, phone) built only from
real contact fields and click-wrapped via `worker_url`. When `worker_url` is empty the
demo MUST render untracked (existing convention).

#### Scenario: Tracked CTAs

- GIVEN `worker_url` is set and the prospect has `telefono` and `whatsapp:true`
- WHEN the demo is generated
- THEN phone/WhatsApp CTAs are click-wrapped and the pixel points at `worker_url` with `tracking_id`

#### Scenario: Empty worker_url

- GIVEN `worker_url` is empty
- WHEN the demo is generated
- THEN CTAs use plain links and no tracking pixel or wrapper is added

### Requirement: Maqueta Framing and Estrateg IA Hyper Sello

The demo MUST be framed as a "maqueta/propuesta" (never as already published) and MUST
show a visually distinctive "sello" (credit block) in the footer stating the page was
generated by a fully automated process by Estrateg IA Hyper AND pitching the breadth of
its services — automated workflows, agentic AI scenarios, and automation of manual tasks
with software and AI. The copy MUST stay short (headline plus one supporting line, never
a long paragraph). The sello's styling MUST make it stand out from surrounding footer
text while adapting to each page's footer design and palette.

#### Scenario: Framing and sello present

- GIVEN any generated demo
- WHEN it is rendered
- THEN maqueta/propuesta framing is visible
- AND the sello names Estrateg IA Hyper, states the fully automated process, and
  mentions automation beyond landing pages (workflows, agentic scenarios, manual-task
  automation with software and AI)
- AND the sello is visually distinct from the surrounding footer content

#### Scenario: Sello adapts to each footer design

- GIVEN two demos with different palettes and footer layouts
- WHEN both are rendered
- THEN each sello uses its own page's palette and design language while remaining
  distinctive, and neither outshines the prospect's own CTAs

### Requirement: Pinned Generation Model

The skill MUST delegate the landing's HTML/CSS/JS code generation to a sub-agent pinned
to model `sonnet` (Claude Sonnet 5), so every demo in every campaign is produced by the
same model class and results remain comparable and measurable.

#### Scenario: Generation delegated to the pinned model

- GIVEN a demo generation run
- WHEN the code-writing step executes
- THEN it runs in a sub-agent invoked with model `sonnet`
- AND the `resumen` records the model used

#### Scenario: Pinned model unavailable

- GIVEN the `sonnet` alias is unavailable in the session
- WHEN generation must proceed
- THEN the skill instructs recording the substitute model in the `resumen` so the run
  is never silently unmeasurable
